Retry & Timing

withRetry makes your code resilient. timed() makes it transparent. Together they turn silent failures into observable, recoverable events.

Exponential backoff — the polite insistence

Imagine calling a busy government office. First try: busy. You wait 5 seconds and try again. Still busy. You wait 10 seconds. Then 20. Then 40. You do not hang up the moment it is busy, but you also do not keep redialing every millisecond and jam the lines.

That is exponential backoff with jitter. withRetry does this automatically for every Helius call.

oliver.mjs — withRetry 
export async function withRetry(fn, {
  maxRetries = 4,
  retryBaseMs = 250
} = {}) {
  for (let i = 0; i <= maxRetries; i++) {
    try { return await fn(); }
    catch (err) {
      const transient =
        /429|503|ECONNRESET|fetch failed/i
        .test(err?.message);
      if (!transient || i === maxRetries) throw err;
      await new Promise(r =>
        setTimeout(r, retryBaseMs * 2 ** i
          + Math.random() * 100));
    }
  }
}
PLAIN ENGLISH

Run this function, and if it fails, retry it. Max retries defaults to 4, initial wait to 250ms.

Loop up to (maxRetries + 1) total attempts.

Try to run the function. If it works, return the result immediately.

If it threw an error — check if the error is a temporary ("transient") network problem.

429 = rate limited, 503 = server overloaded, ECONNRESET = connection dropped. These are all recoverable.

If it is NOT a transient error (e.g., bad API key), or we have run out of retries — throw the error for real.

Wait: attempt 0 waits 250ms, attempt 1 waits 500ms, attempt 2 waits 1000ms, attempt 3 waits 2000ms. The random 0-100ms prevents all retrying clients from thundering back at once.

timed() — the black box recorder

Every call in the Oliver pattern is wrapped in timed(). This function records everything that happened — whether it succeeded, how long it took, and how many bytes came back. If it fails, it catches the error gracefully instead of crashing the whole batch.

oliver.mjs — timed() 
export async function timed(label, fn, credits = 1) {
  const t0 = performance.now();
  try {
    const result = await fn();
    const ms = performance.now() - t0;
    const bytes = JSON.stringify(result).length;
    return { label, status: "ok", result, ms, bytes, credits };
  } catch (err) {
    const ms = performance.now() - t0;
    return { label, status: "err", error: err?.message, ms };
  }
}
PLAIN ENGLISH

Run function fn, label the result as label, track how many API credits it costs.

Note the time right now — this is our starting stopwatch.

Try to run the function...

If it worked, record the result.

Calculate how many milliseconds the call took.

Measure the response size by serializing it to text and measuring character count.

Return a success object with everything we know about this call.

If it threw an error — do NOT let it crash the whole batch. Catch it, record the elapsed time and error message, return an error object instead.

Why "never crash the batch" is a superpower

Without timed(), if one of your 9 Helius calls fails, the entire Promise.all crashes and you lose all the data from the other 8 successful calls. With timed(), every call returns a result object — either status: "ok" or status: "err". You get partial results instead of nothing.

status: "ok" Call succeeded. result, ms, bytes, credits are all populated.
status: "err" Call failed. error message is populated. ms tells you how long it took to fail. Batch continues without crashing.
credits How many Helius credits this call cost. Summed across all calls to show your total monthly burn rate.
bytes Size of the response payload in bytes. Useful for spotting unexpectedly large (or suspiciously empty) responses.
💡
Error objects over thrown exceptions — a production pattern

In production systems, it is often better to return a structured error object than to throw an exception. Throwing crashes the whole operation. Returning an error object lets the rest of the system continue — and gives you something to display to the user or log for debugging. You will see this pattern in every large-scale system.
