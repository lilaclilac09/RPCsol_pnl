The Semaphore

12 callers, one bottleneck, zero crashes. How makeSemaphore keeps you off the Helius ban list.

A highway on-ramp, not a traffic jam

Imagine a highway where only 12 cars are allowed on at a time. When a 13th car arrives, it waits at the on-ramp signal. The moment one car exits, the signal turns green for the next waiting car.

That is a semaphore. In oliver.mjs, the highway is the Helius API, the 12-car limit is the MAX_CONCURRENCY = 12 constant, and every API call must take its place in the queue.

⚠️
Why this matters on Helius free tier

Helius rate-limits aggressive clients. If you fire 50 simultaneous requests, you will get a 429 error and potentially a temporary ban. The semaphore caps you at 12 in-flight at any moment — fast enough to feel instant, safe enough to avoid the ban.

The 6 lines that make it work

Here is the complete semaphore from oliver.mjs. It is 6 lines of code, but it contains one genuinely clever trick:

oliver.mjs 
export function makeSemaphore(limit) {
  let active = 0;
  const queue = [];
  return () => new Promise(resolve => {
    const go = () => { active++; resolve(() => { active--; queue.shift()?.(); }); };
    active < limit ? go() : queue.push(go);
  });
}
PLAIN ENGLISH

makeSemaphore(limit) — create a gate that allows at most limit things through at once.

active — a counter: how many requests are currently in-flight right now.

queue — a waiting room. Requests that cannot go yet sit here until a slot opens.

return () => new Promise — the function you call to "ask permission". It gives you back a promise that resolves when it is your turn.

go() — the green light. Increment active (occupy a slot), then hand you a "release" function to call when done. The release function decrements active and lets the next queued request in.

active < limit ? go() : queue.push(go) — if there is room, go immediately. Otherwise, get in line.

Acquire and release in action

Every single Helius call in oliver.mjs follows this exact pattern. Notice how the finally block guarantees the slot is always released — even if the network call crashes.

oliver.mjs — rpcPost() 
async function rpcPost(method, params) {
  return withRetry(async () => {
    const release = await acquire();
    try {
      const r = await fetch(RPC_URL, { ... });
      return await r.json();
    } finally { release(); }
  });
}
PLAIN ENGLISH

This function makes one Solana RPC call and returns the result.

Wrap the whole thing in withRetry so network glitches are handled automatically.

await acquire() — wait at the on-ramp until a slot is available. When you get the green light, acquire() gives you back a "release" function.

Now inside the slot — make the actual HTTP request to Helius.

Get the response JSON and return it.

finally { release() } — ALWAYS free up the slot when done, whether the call succeeded or crashed. This is critical — without it, slots would leak and the semaphore would eventually block everything.

💡
The try/finally pattern is everywhere in production code

Any time you "take" a resource (a database connection, a file lock, a semaphore slot), you must return it when done — even if something goes wrong. The try/finally block guarantees this. This is one of those patterns engineers use instinctively. Now you know why.
