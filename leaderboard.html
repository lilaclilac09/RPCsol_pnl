Per-Algorithm Failure Modes
Fixed bugs, active risks, and known edge cases for each algorithm.

V1 — Golomb Probing

✗
Window overflow on dense wallets
Density estimation via geometric probing is approximate. On burst-activity wallets (many txns in narrow slot range), windows overflow → extra serial pagination → O(N) worst case.
✗
Refinement thrashing
Water-fill refinement rounds re-probe partially filled windows. Poorly estimated density triggers exponential call growth in extreme cases.
!
Budget counter bug (fixed in research)
Original code checked rpcCalls (window count) not rpc.callCount() (actual API calls). Caused 133 calls on dense wallet instead of 40.

V2 — Signatures-First

✗
3 serial round-trips minimum
Anchor (2 calls) → sig enumeration → parallel windows. Each is a serial dependency. Minimum latency is bounded by 3 × RTT regardless of parallelism.
✗
Sequential sig pagination for >1000 txn wallets
Every 1000 additional transactions adds 1 more serial round-trip (1 sig page). A 5000-txn wallet adds 4 extra serial RTs.
!
Anchor size hard cap at 100
Helius API returns max 100 full txns per call regardless of limit value. Setting anchorSize > 100 silently truncates.

V3 — Merged Anchor-Sig

✓
Helius paginationToken quirk (fixed)
API returns paginationToken even when results < limit. Now treated as truncated only when results.length ≥ sigPageSize.
✓
Slot filter gte:0 rejected (fixed)
When anchor returns zero samples (all zero-delta), oldestLastSlot = −1 → gapFrom = 0 → Helius rejects gte:0. Fixed: clamp to max(1, oldestLastSlot+1).
✓
Inverted slot range (fixed)
When anchor covers up to the newest sig, gapFrom > maxSlot − 1 → invalid range. Fixed: early-return when gapFrom ≥ maxSlot.
✗
Active: large-wallet serial fallback (Case 4)
Wallets with >1000 txns fall back to sequential sig pagination. V4 eliminates this with a single density probe.

V4 — Hybrid Density

!
Density overestimate → too many windows
The 3-point density estimate (oldest, newest, probe spans) takes the max — conservative by design. On burst wallets, this fires too many small windows. Same failure as Codex on the dense test wallet, but less severe because V4 is exact for <1000 sigs.
!
blockTime null on some transactions
Some historical Solana txns have blockTime: null. blockTime-based window queries may miss or mis-order these. Affects <0.1% of mainnet transactions.
!
Extra probe call for all >1000 txn wallets
V4 always fires 1 density probe call for large wallets, even if the wallet is borderline (1001 txns). Still 2 serial RTs total, but 1 extra concurrent call in Phase 1.

Codex — blockTime Density (sol_pnl.ts)

✗
Over-partitioning on dense wallets
Default density estimation fires 123 calls on the dense test wallet vs 12 for V3/V4. Conservative max-density approach creates many tiny time windows.
✓
Avoids slot-filter quirks by design
Uses blockTime filters throughout — no gte:0 or inverted-range bugs. Correct for very old (pre-2021) transactions where slot numbers are unreliable.
!
Unoptimized — no hyperparameter search run yet
Only tested with default config (windowTarget=80, c=12). BO/TPE on Codex variant with tuned windowTarget could close the gap significantly.

Production Recommendation
Choose based on your wallet size distribution and API tier.

✅ Recommended: V4 — All Wallet Sizes

Use when wallet transaction history is unknown or may exceed 1000 txns.
import { solBalanceOverTime } from "./sol_balance_v4.mjs";

const strategy = {
  anchorSize:     92,   // full txns in oldest anchor
  windowTarget:   79,   // txns per fetch window
  windowSize:     79,   // same for slot-based path
  sigPageSize:   1000,
  maxConcurrency:  9,
  skipZeroDelta: false,  // include all txns
};

const result = await solBalanceOverTime(
  address, apiKey, strategy
);
// result.points[]  ← chronological SOL balance
// result.stats     ← totalRpcCalls, wallTimeMs
  • 2 serial round-trips for ALL wallet sizes — no sequential sig pagination fallback
  • 72% faster than V2 default config (1267ms vs 3650ms avg)
  • Handles wallets with >1000 txns via a single blockTime density probe

Fastest: V3 — Known Small Wallets

Use when you know wallets have <1000 succeeded transactions.
import { solBalanceOverTime } from "./sol_balance_v3.mjs";

const strategy = {
  anchorSize:    76,
  windowSize:    52,
  sigPageSize:  1000,
  maxSigPages:    20,
  maxConcurrency: 23,  // higher than V2!
  skipZeroDelta: false,
};

const result = await solBalanceOverTime(
  address, apiKey, strategy
);
  • Peak score 0.879 — fastest observed latency ~1138ms on all 3 wallets
  • c=23 optimal — 2-RT structure means parallel windows benefit from high concurrency
  • Falls back to sequential sig pagination for >1000 txn wallets

Key Insights
Non-obvious findings from 200+ evaluated configurations across 7 search methods.
V8

V12 Score-Max: 0.744 score, +81.8% vs V3. Window=105 eliminates per-window pagination.

V10 fast-exit (anchor=100 always): sparse 330ms / 1 RT. V12 Score-Max (50-trial BO, window=105): sparse 314ms, medium 602ms, dense 3117ms, avg 1344ms. Key insight: window>100 fits all dense-wallet transactions in one fullTxns page — no serial pagination within each window. V12 Latency-Min (bottleneck objective) minimizes worst-case per-wallet time (maxMs=3673ms). V8 added streaming; V10 added fast-exit; V12 expanded search to window [50,200] and found the no-pagination sweet spot at 105.

RT

Optimal concurrency depends on the algorithm's round-trip count

V2 optimal: c=6 (3 serial RTs — rate-limit retries dominate). V3 optimal: c=23 (2 serial RTs — Phase 1 parallel windows are the bottleneck). V4 optimal: c=9 (density windows are larger, less parallelism needed).

56%

2 serial round-trips beats 3 by ~56% on average

V3's key innovation: firing oldest full txns + newest sigs in a single Promise.all removes one complete serial dependency. For the medium wallet, Phase 0 alone delivers all data — Phase 1 fires 2–3 windows in parallel.

VAR

Score variance is too high for single-trial comparison

Helius free tier has ~2x latency jitter on the sparse wallet (871ms–2970ms). A single BO trial score of 0.879 vs a re-evaluated 0.68 illustrates the noise. Use 3-run averages for stable comparison.

DE

Differential Evolution found the best V2 config in 8 evaluations

DE/rand/1/bin reached score 0.620 in generation 1 (8 pop x 1 gen). This beat GP-UCB (0.458 at trial 15) by 35%. DE's population crossover naturally explores oracle=true territory that sequential search misses.

HB

Hyperband failed due to poor fidelity correlation

The sparse wallet always fast-exits in 2 calls regardless of config — it's not a discriminating signal. Low fidelity (sparse only) filters out good configs by chance. Fix: use dense wallet as the only low-fidelity proxy.

API

Helius API has undocumented quirks that break naive implementations

1. Returns paginationToken even when results < limit. 2. Rejects slot filters with gte:0. 3. Hard caps full transaction responses at 100 regardless of limit param. All three were hit and fixed.

SKP

skipZeroDelta=false consistently reduces overhead

Filtering transactions where pre===post adds a comparison loop over every transaction. For our test wallets (~14–22 total RPC calls), this loop runs thousands of times. The filter cost exceeds the benefit of excluding zero-delta txns.

BT

blockTime vs slot filtering: different tradeoffs

Slot-based (V2/V3/V4): faster exact windows, but triggers Helius slot-filter quirks. blockTime-based (Codex): immune to quirks, richer output, but density estimation can over-partition on dense wallets.
