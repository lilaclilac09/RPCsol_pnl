The Scout Algorithm

The three-phase strategy that turns a potential 45-second crawl into a 300ms sprint

The expensive problem

Fetching a Solana wallet's full transaction history is expensive. Not in money — in time. Each transaction requires a network request, and a request takes 50–150ms. A wallet with 5000 transactions fetched one-by-one would take 4–12 minutes.

⚠️
The naive approach fails completely

Fetch transaction 1, wait. Fetch transaction 2, wait. Fetch transaction 3, wait... 5000 transactions × 100ms = 8 minutes. Useless for any real application.

The Scout algorithm solves this by asking a smarter question: "What's the minimum number of requests I need to make, in the most parallelized order possible?"

💡
The key insight

Solana's RPC API offers two modes: signature-only (cheap — 10× smaller response, 1000 results per request) and full transaction (expensive — complete data). Scout uses signatures to plan before committing to expensive full fetches.

Three phases — each building on the last

Scout doesn't just "fetch everything." It runs a three-phase reconnaissance mission before committing to expensive work.

0

Phase 0 — Anchor (1 round trip)

Fire exactly 2 requests simultaneously: oldest 1000 signatures + newest 1000 signatures. This anchors the wallet's full time range and tells us the total transaction count — all in a single RTT.

1

Phase 1 — Scout (adaptive)

Only runs if the wallet has more than 2000 transactions. Splits the middle slot range into 4–12 adaptive partitions and fires signature-only requests in parallel. This maps the density of transactions across time — cheap, fast reconnaissance.

2

Phase 2 — Full Fetch (parallelized)

Now that Scout knows exactly where the transactions are, it fires full-mode requests in ~50-signature chunks, streaming results as each partition completes. Up to 100 requests in flight simultaneously.

The 10× cheaper trick

The most clever part of Scout is that it never pays full price until it knows exactly what to buy.

❌ Full Transaction Request

  • 100 transactions per request
  • Full JSON blob per tx
  • Pre/post balances, logs, instructions
  • ~50KB per request
  • 1 API credit per transaction

✓ Signature-Only Request

  • 1000 results per request
  • Just the signature + slot
  • Minimal data per entry
  • ~5KB per request
  • Equivalent to 100 full-tx API cost

By scouting with signatures first, Scout can survey 1000 transaction positions for the same cost as 100 full fetches. It uses this cheap data to plan the expensive phase precisely.

The algorithm in code

Here's the heart of Phase 0 — the "anchor" step that fires two parallel requests to bound the wallet's range:

algorithm.rs 
// Phase 0: anchor oldest + newest in one RTT
let (oldest, newest) = tokio::join!(
    rpc.fetch_signatures(address, oldest_params),
    rpc.fetch_signatures(address, newest_params),
);
PLAIN ENGLISH

Comment: Phase 0 — get both ends of the wallet history in one round trip

tokio::join! — start BOTH requests simultaneously, wait for BOTH to finish

Request the oldest 1000 signatures (first-ever transactions)

Request the newest 1000 signatures (most recent transactions)

Both results arrive together — we now know the wallet's full time span

💡
Async/await — the secret to concurrency

Async code lets the program start a request, immediately start another request, and only wait when it actually needs the result. This is how 100 requests run in parallel without 100 threads.
