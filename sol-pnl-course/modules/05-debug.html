When Things Break

Debugging intuition, common failure modes, and how to extend this codebase confidently

Every failure has a home

When this tool breaks, the error almost always originates in one specific layer. Knowing the map saves you from searching blindly.

🔑

Bad API Key / No Key

Error from rpc.rs immediately. Helius returns 401 Unauthorized. The retry loop won't help — a bad key is a bad key.

🌐

Network / Timeout

rpc.rs retries up to 5 times with backoff. If all 5 fail, you'll see an error indicating connection failure. Check your internet connection or Helius status.

🏷️

Invalid Wallet Address

Helius returns an empty result or an error. The algorithm sees zero transactions and returns zero PnL. Always double-check the address format.

🚦

Rate Limit (429)

Even with the semaphore, sustained heavy use can trigger rate limits. The retry logic handles brief spikes — but sustained abuse will exhaust all 5 retries.

Debugging by layer

Like a detective who starts at the scene and works backward, debugging this tool means finding which layer produced the symptom.

SYMPTOM → LAYER 
// Symptom: "tx_count is 0, pnl is 0"
// Likely: bad wallet address or empty wallet
// Check: algorithm.rs Phase 0 result

// Symptom: "error after a few seconds"
// Likely: network or API key issue
// Check: rpc.rs fetch_signatures return value

// Symptom: "wrong tx count on large wallet"
// Likely: deduplication edge case in Phase 2
// Check: algorithm.rs signature dedup logic
PLAIN ENGLISH

If you see zero transactions returned...

The wallet address was wrong or it has never transacted

Investigate: add a print statement after Phase 0 to see raw results

 

If the tool crashes with an error after trying for a few seconds...

Something is wrong with the connection or authentication

Investigate: look at rpc.rs to see what error code it received

 

If the count is slightly off for big wallets...

A transaction may have been counted twice (boundary overlap)

Investigate: the deduplication step in algorithm.rs

The hardest edge case: Address Lookup Tables

Solana v0 transactions use a compression technique called Address Lookup Tables — they compress transaction data by referencing accounts by index rather than full address. rpc.rs handles this explicitly.

rpc.rs — extract_balance() 
// Combine static keys with ALT keys
let all_keys: Vec<String> = tx
    .transaction.message
    .accountKeys
    .iter()
    .chain(loaded_addresses)
    .cloned()
    .collect();
PLAIN ENGLISH

Comment: merge both types of account key into one list

Start with the transaction's main key list

...which is inside the message part of the transaction

Get the built-in ("static") account keys

Turn it into an iterator so we can chain onto it

Chain on the Address Lookup Table keys (dynamic, loaded separately)

Make copies of each key string

Collect everything into a single combined list

Without this, v0 transactions would fail to find the wallet address in the account list and silently return no balance data. This is the kind of edge case that only surfaces with real-world wallets.

Extending the tool with confidence

Now that you understand the architecture, here's how to describe changes to an AI coding agent precisely — without ambiguity.

1

Adding a new output field

Say: "In types.rs, add a fee_lamports: u64 field to the BalancePoint struct. Then in rpc.rs extract_balance(), read the fee from the transaction's meta.fee field and populate it."

2

Supporting a different wallet standard

Say: "In rpc.rs, add a new method alongside fetch_page() that calls the getTokenAccountsByOwner endpoint. Keep the same retry and semaphore pattern."

3

Changing the output format

Say: "In main.rs, after we compute the BalancePoints, instead of printing JSON, write the data to a CSV file with columns: slot, signature, pre_sol, post_sol, delta_sol."

The full picture

You've now seen the entire codebase from the outside in. Here's what you've learned — and why it matters for everything you build next.

🏗️

Single responsibility = easy AI prompting

Because each file has one job, you can tell an AI exactly which file to touch. No guessing, no "do it in the right place."

🔄

Retry logic is load-bearing

Any new network call you add should use the same 5-try exponential backoff pattern. Ask for it explicitly when prompting AI.

⚡

The speed secret: plan cheaply, execute precisely

Scout's three phases are a general pattern: survey cheaply, plan precisely, execute in parallel. This pattern applies to any API that has cheap metadata endpoints and expensive full-data endpoints.
