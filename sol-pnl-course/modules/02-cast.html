Meet the Cast

Four Rust files, four distinct personalities — and how they divide the work

Think of it like a film production

Every professional film has a clear division of labor. Nobody asks the cinematographer to also write the script. The codebase works the same way — each file has one job and stays in its lane.

🎬

main.rs

The Director — reads your command, coordinates all the other parts, and delivers the final output

🧠

algorithm.rs

The Strategist — decides the optimal plan for fetching data and assembles the final timeline

📡

rpc.rs

The Field Reporter — makes the actual network calls to Helius and returns raw data

📦

types.rs

The Prop Master — defines the shapes of all data (the structs that hold each balance point)

The codebase at a glance

The entire project is just four source files plus a Cargo.toml. Elegant simplicity.

📁 sol-pnl-challenge/
├── 📁 src/
├── main.rs — entry point, CLI parsing, output
├── algorithm.rs — Scout algorithm, 3-phase strategy
├── rpc.rs — HTTP client, Helius API calls
└── types.rs — BalancePoint struct, data shapes
└── Cargo.toml — dependencies (reqwest, tokio, serde…)
💡
The "single responsibility" principle

Each file does exactly one thing. This is a core software engineering principle — and it's what makes AI good at modifying code. When you tell an AI "update how the HTTP client works," it knows to look in rpc.rs and nowhere else.

The data shape everything revolves around

types.rs defines BalancePoint — the single unit of data representing one transaction's effect on a wallet's balance. Every transaction becomes one of these.

types.rs 
pub struct BalancePoint {
    pub slot: u64,
    pub block_time: Option<i64>,
    pub pre_lamports: u64,
    pub post_lamports: u64,
    pub signature: String,
}
PLAIN ENGLISH

A struct called BalancePoint — one snapshot of a transaction

slot: Solana's version of a block number — which moment in time this happened

block_time: The Unix timestamp (seconds since 1970). Option means it might be missing

pre_lamports: Balance BEFORE this transaction (in lamports)

post_lamports: Balance AFTER this transaction (in lamports)

signature: The unique ID of this transaction on the blockchain

algorithm.rs

4821 total — that means 2821 more in the middle. I'll partition the remaining slot range into 6 chunks. rpc.rs, fetch all 6 in parallel with full transaction data.

R
rpc.rs

Sending up to 100 concurrent requests... All 4821 transactions received. Passing raw data back.

A
algorithm.rs

Converting each transaction into a BalancePoint using the types.rs struct. Deduplicating, sorting by slot... main.rs, here are 4821 clean BalancePoints.

M
main.rs

Got it. Calculating PnL (last balance minus first balance), measuring total latency... Printing JSON output. Done in 312ms! ✓
