The world this tool lives in

To understand what this tool does, you need to know two things about Solana.

1

Everything is public

Every transaction ever made on Solana is recorded on the blockchain — forever, publicly. Anyone with a wallet address can reconstruct the full financial history of that wallet. There's no login required.

2

Wallets have no "balance field"

Your wallet balance isn't stored anywhere. It's calculated by reading every transaction that ever touched your address and summing the changes. This tool does that calculation — for every transaction, in order.

💡
Why this is hard

A popular wallet might have 50,000+ transactions. Each one requires a network request. Fetching them one-by-one would take minutes. This tool's entire job is to make that fast — and it does it in under a second.

The data shape everything revolves around

types.rs defines BalancePoint — the single unit of data representing one transaction's effect on a wallet's balance. Every transaction becomes one of these.

types.rs 
pub struct BalancePoint {
    pub slot: u64,
    pub block_time: Option<i64>,
    pub pre_lamports: u64,
    pub post_lamports: u64,
    pub signature: String,
}
PLAIN ENGLISH

A struct called BalancePoint — one snapshot of a transaction

slot: Solana's version of a block number — which moment in time this happened

block_time: The Unix timestamp (seconds since 1970). Option means it might be missing

pre_lamports: Balance BEFORE this transaction (in lamports)

post_lamports: Balance AFTER this transaction (in lamports)

signature: The unique ID of this transaction on the blockchain

The Scout Algorithm

The three-phase strategy that turns a potential 45-second crawl into a 300ms sprint

The expensive problem

Fetching a Solana wallet's full transaction history is expensive. Not in money — in time. Each transaction requires a network request, and a request takes 50–150ms. A wallet with 5000 transactions fetched one-by-one would take 4–12 minutes.

⚠️
The naive approach fails completely

Fetch transaction 1, wait. Fetch transaction 2, wait. Fetch transaction 3, wait... 5000 transactions × 100ms = 8 minutes. Useless for any real application.

The Scout algorithm solves this by asking a smarter question: "What's the minimum number of requests I need to make, in the most parallelized order possible?"

💡
The key insight

Solana's RPC API offers two modes: signature-only (cheap — 10× smaller response, 1000 results per request) and full transaction (expensive — complete data). Scout uses signatures to plan before committing to expensive full fetches.

Three phases — each building on the last

Scout doesn't just "fetch everything." It runs a three-phase reconnaissance mission before committing to expensive work.

0

Phase 0 — Anchor (1 round trip)

Fire exactly 2 requests simultaneously: oldest 1000 signatures + newest 1000 signatures. This anchors the wallet's full time range and tells us the total transaction count — all in a single RTT.

1

Phase 1 — Scout (adaptive)

Only runs if the wallet has more than 2000 transactions. Splits the middle slot range into 4–12 adaptive partitions and fires signature-only requests in parallel. This maps the density of transactions across time — cheap, fast reconnaissance.

2

Phase 2 — Full Fetch (parallelized)

Now that Scout knows exactly where the transactions are, it fires full-mode requests in ~50-signature chunks, streaming results as each partition completes. Up to 100 requests in flight simultaneously.

The 10× cheaper trick

The most clever part of Scout is that it never pays full price until it knows exactly what to buy.

❌ Full Transaction Request

  • 100 transactions per request
  • Full JSON blob per tx
  • Pre/post balances, logs, instructions
  • ~50KB per request
  • 1 API credit per transaction

✓ Signature-Only Request

  • 1000 results per request
  • Just the signature + slot
  • Minimal data per entry
  • ~5KB per request
  • Equivalent to 100 full-tx API cost

By scouting with signatures first, Scout can survey 1000 transaction positions for the same cost as 100 full fetches. It uses this cheap data to plan the expensive phase precisely.

The algorithm in code

Here's the heart of Phase 0 — the "anchor" step that fires two parallel requests to bound the wallet's range:

algorithm.rs 
// Phase 0: anchor oldest + newest in one RTT
let (oldest, newest) = tokio::join!(
    rpc.fetch_signatures(address, oldest_params),
    rpc.fetch_signatures(address, newest_params),
);
PLAIN ENGLISH

Comment: Phase 0 — get both ends of the wallet history in one round trip

tokio::join! — start BOTH requests simultaneously, wait for BOTH to finish

Request the oldest 1000 signatures (first-ever transactions)

Request the newest 1000 signatures (most recent transactions)

Both results arrive together — we now know the wallet's full time span

💡
Async/await — the secret to concurrency

Async code lets the program start a request, immediately start another request, and only wait when it actually needs the result. This is how 100 requests run in parallel without 100 threads.

The Messenger

How rpc.rs talks to Helius — HTTP/2, JSON-RPC, concurrency, and bulletproof retry logic

The language of blockchain APIs

To get data from Helius, you don't visit a website — you send carefully formatted messages using a protocol called JSON-RPC.

Think of it like a postal service with a strict form template. Every letter (request) must follow the same format or it gets returned unread.

JSON-RPC REQUEST 
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "getSignaturesForAddress",
  "params": [
    "7xK...mNp",
    { "limit": 1000 }
  ]
}
PLAIN ENGLISH

Opening the message envelope

Using the standard JSON-RPC version 2.0 format

An ID to match this request with its response later

The function name we want to call on the Helius server

The arguments — starting with...

The wallet address to look up

And optional settings: return up to 1000 results

Close the message

HTTP/2 — the highway upgrade

Most web requests use HTTP/1.1, which is like a single-lane road: one car at a time. This tool uses HTTP/2, which is like a multi-lane highway.

HTTP/1.1 — Serial

Request 1 ████████
Request 2 ████████
Request 3 ████████

Total time: 3× per-request time

HTTP/2 — Parallel

Request 1 ████████
Request 2 ████████
Request 3 ████████

Total time: 1× per-request time ⚡

The tool opens one TLS connection to Helius and sends up to 100 requests through it simultaneously. That single connection has a massive flow-control window to avoid bottlenecks.

The 100-request speed limit

More parallel requests = more speed, right? Not always. Flood Helius with 5000 simultaneous requests and they'll rate-limit you. The tool uses a semaphore to cap at exactly 100 concurrent requests.

rpc.rs 
// 100-concurrent-request semaphore
let sem = Arc::new(
    Semaphore::new(100)
);
// Each request acquires one permit
let _permit = sem.acquire().await;
PLAIN ENGLISH

Comment: we limit ourselves to 100 requests at once

Arc::new — create a shared semaphore that all tasks can access safely

Set the capacity to 100 — like a parking lot with 100 spaces

Close the bracket

Comment: before sending a request, grab a permit

Wait for a free slot, then claim it — when this task finishes, the slot is released automatically

Retry logic — never give up (gracefully)

Networks are unreliable. Helius sometimes returns errors. The messenger never crashes on a single failure — it uses exponential backoff to retry up to 5 times.

1

Try 1 — immediate

Send the request. If we get a 429 or 5xx error, don't panic — try again.

2

Try 2 — wait 100ms

Brief pause, then retry. The server might have just been momentarily overloaded.

3–5

Tries 3–5 — growing delays

Each retry waits twice as long: 200ms → 400ms → 800ms. By try 5, we've given the server real time to recover. If all 5 fail, then — and only then — return an error.

💡
Why this matters when you extend the tool

If you ask an AI to add a new API call to this codebase, make sure it uses the same retry pattern. A missing retry on a new function will cause flaky failures that are hard to debug — the kind of bug where "it works most of the time."

When Things Break

Debugging intuition, common failure modes, and how to extend this codebase confidently

Every failure has a home

When this tool breaks, the error almost always originates in one specific layer. Knowing the map saves you from searching blindly.

🔑

Bad API Key / No Key

Error from rpc.rs immediately. Helius returns 401 Unauthorized. The retry loop won't help — a bad key is a bad key.

🌐

Network / Timeout

rpc.rs retries up to 5 times with backoff. If all 5 fail, you'll see an error indicating connection failure. Check your internet connection or Helius status.

🏷️

Invalid Wallet Address

Helius returns an empty result or an error. The algorithm sees zero transactions and returns zero PnL. Always double-check the address format.

🚦

Rate Limit (429)

Even with the semaphore, sustained heavy use can trigger rate limits. The retry logic handles brief spikes — but sustained abuse will exhaust all 5 retries.

Debugging by layer

Like a detective who starts at the scene and works backward, debugging this tool means finding which layer produced the symptom.

SYMPTOM → LAYER 
// Symptom: "tx_count is 0, pnl is 0"
// Likely: bad wallet address or empty wallet
// Check: algorithm.rs Phase 0 result

// Symptom: "error after a few seconds"
// Likely: network or API key issue
// Check: rpc.rs fetch_signatures return value

// Symptom: "wrong tx count on large wallet"
// Likely: deduplication edge case in Phase 2
// Check: algorithm.rs signature dedup logic
PLAIN ENGLISH

If you see zero transactions returned...

The wallet address was wrong or it has never transacted

Investigate: add a print statement after Phase 0 to see raw results

 

If the tool crashes with an error after trying for a few seconds...

Something is wrong with the connection or authentication

Investigate: look at rpc.rs to see what error code it received

 

If the count is slightly off for big wallets...

A transaction may have been counted twice (boundary overlap)

Investigate: the deduplication step in algorithm.rs

The hardest edge case: Address Lookup Tables

Solana v0 transactions use a compression technique called Address Lookup Tables — they compress transaction data by referencing accounts by index rather than full address. rpc.rs handles this explicitly.

rpc.rs — extract_balance() 
// Combine static keys with ALT keys
let all_keys: Vec<String> = tx
    .transaction.message
    .accountKeys
    .iter()
    .chain(loaded_addresses)
    .cloned()
    .collect();
PLAIN ENGLISH

Comment: merge both types of account key into one list

Start with the transaction's main key list

...which is inside the message part of the transaction

Get the built-in ("static") account keys

Turn it into an iterator so we can chain onto it

Chain on the Address Lookup Table keys (dynamic, loaded separately)

Make copies of each key string

Collect everything into a single combined list

Without this, v0 transactions would fail to find the wallet address in the account list and silently return no balance data. This is the kind of edge case that only surfaces with real-world wallets.
